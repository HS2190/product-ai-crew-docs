# Product AI Crew

> 리서처 · PM · 기획자 · 디자이너 · UX라이터 · Engineer, 그리고 매 단계를 검수하는 Reviewer로 구성된 AI 멀티 에이전트 제품 조직

---

## 한 줄 소개

조직에 없는 역할을 AI 에이전트로 채워, 디자이너 혼자서도 리서치부터 **동작하는 프론트엔드 구현까지** 전체 프로덕트 파이프라인을 돌릴 수 있는 시스템입니다.

> UX 디자이너가 **AI를 실무에 활용하기 위해** 직접 설계하고 운영하는 작업 시스템입니다. 챌린지 제출작·실험 프로젝트를 실제로 이 파이프라인으로 만들면서 다듬어 왔습니다.

---

## 만든 이유

### 디자인 업무를 하면서 느낀 역할 공백

UX 디자이너로 일하면서 "이 기능을 왜 만들어야 하는가"를 정의하는 PM도, 화면 흐름을 체계적으로 기획하는 기획자도 없는 상황을 자주 마주쳤습니다. 결국 디자인 이전 단계부터 모두 혼자 정리해야 했습니다.

### UX 라이팅 봇에서 시작해 파이프라인 전체로

처음에는 Claude Code로 UX 라이팅 전용 에이전트를 만들어 썼습니다. 반복 작업이 눈에 띄게 줄어드는 걸 경험하면서 "그 위 단계들(기획·PM·리서치)도, 그 아래 단계(구현)도 AI로 채울 수 있지 않을까?"로 생각이 확장됐습니다.

### 채우다 보니 '조직'이 필요해졌다

역할을 하나씩 채우자 새로운 문제가 보였습니다. **누가 각 산출물의 품질을 봐주는가**, 그리고 **한 번 내린 판단을 다음 프로젝트가 어떻게 기억하는가**. 그래서 단순한 에이전트 나열을 넘어 두 가지를 더했습니다 — 매 단계 산출물을 비평하는 **Reviewer 검수 루프**, 그리고 프로젝트를 넘어 판단·선호를 쌓는 **기억층**. 각 역할의 인풋/아웃풋과 검수·기억 흐름을 명확히 정의하니 AI 에이전트도 협업 동료처럼 동작했습니다.

## 시스템 구조

```text
[사용자 요청]
      │
      ▼
[오케스트레이터] ── 모드 판단 · 에이전트 순서 결정 · 세션 관리
      │            페르소나 + 기억층(회사별 결정·컨벤션)을 모든 단계에 자동 주입
      ▼
[Researcher] → [PM] → [기획자] → [디자이너] → [UX라이터] → [Engineer]
   리서치 종합    PRD    화면 기획    디자인 스펙    화면 문구    동작 코드 ★최종 산출물
      │            │        │            │            │            │
      └──── 각 단계 산출물마다 ─[ Reviewer 검수 ]─ pass / revise(최대 2회) / escalate
                                         │
                                         ▼
                            [작업 완료 — 동작하는 프론트엔드]
```

- **최종 산출물은 Engineer의 동작하는 프론트엔드 코드**입니다. (기획·디자인 문서가 아니라 실제로 돌아가는 화면까지가 범위입니다. 백엔드·API는 범위 밖으로, 데이터는 mock으로 처리합니다.)
- 각 단계 산출물은 다음 단계로 넘어가기 전 **존재 검증 → Reviewer 검수** 두 게이트를 통과해야 합니다.

## 구성된 에이전트

| 에이전트 | 역할 | 주요 산출물 |
| --- | --- | --- |
| **Researcher** | 리서치 수집·종합, 경쟁 분석, insights 도출 | 리서치 종합, 경쟁 분석, key insights |
| **PM** | 제품 전략, 기능 우선순위(Impact×Effort) 결정 | PRD, 기능 목록, 우선순위, 브랜드 방향성 |
| **기획자** | 화면 기획, User Flow·IA 설계, 상태 정의 | 화면 기획안, 기능 명세서, 와이어프레임 |
| **디자이너** | UI 디자인, 컴포넌트 스펙, 디자인 토큰 | 디자인 스펙, 컴포넌트 스펙, (조건부) Figma |
| **UX 라이터** | UI 문구 설계, 보이스·톤, 금지 표현 관리 | 라이팅 가이드, 화면별 문구 시트 |
| **Engineer** | 프론트엔드 구현, 빌드·접근성 검증 | 동작하는 코드, 빌드 결과 *(최종 산출물)* |
| **Reviewer** | 각 단계 산출물의 내용 품질을 비평하는 검수 레이어 | pass / revise / escalate 판정 |

> Reviewer는 무언가를 *만드는* 역할이 아니라, 생성 에이전트의 산출물이 다음 단계로 넘어가기 전에 내용 품질을 *비평*하는 read-only 레이어입니다. 같은 산출물은 최대 2회까지 재작업(`revise`)하고, 그래도 미흡하면 `escalate`로 사용자 판단에 넘겨 무한 루프를 막습니다.

---

## 이 시스템을 다르게 만드는 세 가지

### 1. 검수 루프 — 산출물마다 비평이 붙는다

생성 에이전트가 산출물을 내면, 다음 단계로 넘어가기 전에 Reviewer가 내용 품질을 비평합니다. `pass`면 진행, `revise`면 검수 의견과 함께 재작업(최대 2회), `escalate`면 사용자 판단으로 넘깁니다. "그럴듯하지만 약한 산출물"이 다음 단계로 새는 것을 막는 장치입니다.

### 2. 기억층 — 쓸수록 똑똑해진다

프로젝트를 *넘어* 누적되는 맥락을 담는 2층 구조입니다.

| 층 | 담는 것 |
| --- | --- |
| **개인 프로파일** | 회사가 바뀌어도 유지되는 나의 작업 취향·원칙 |
| **회사별 결정·컨벤션** | 그 회사에서 내린 판단의 근거(왜) · 톤 · 규칙 · 제약 |

작업 시작 시 해당 회사의 기억을 **자동으로 모든 에이전트에 주입**하고, 의미 있는 판단이 나오면 **제안 → 승인**을 거쳐서만 누적합니다(자동 저장으로 쓰레기가 쌓이는 것을 방지). 회사별로 분리해 A사의 결정이 B사로 새지 않습니다.

> 같은 서비스를 두 번 실행하며 **기억 자동 주입과 누적이 실제로 작동하는 것을 확인**했습니다(2차 실행 시 1차의 결정·컨벤션이 주입되어 적용됨). 다만 아직 소수 프로젝트 규모의 검증입니다.

### 3. 디자인 스킬 선택 게이트

디자이너 단계에서, 에이전트가 기획 결과물을 분석해 이 프로젝트에 맞는 디자인 스타일/시스템 스킬을 **근거와 함께 추천**합니다. 라벨 동어반복이 아니라 기획의 구체적 특징("색=의미 희소자원·고밀도 정보위계" 같은)에 기반한 추천이며, **사용자가 고를 때까지 멈춥니다**. 승인된 스킬을 본작업에 강제 적용해, 매번 빈손에서 토큰을 새로 만드는 것을 막습니다.

## 실행 모드

| 모드 | 에이전트 순서 | 사용 상황 |
| --- | --- | --- |
| **FULL** | Researcher → PM → 기획자 → 디자이너 → UX라이터 → Engineer | 신규 서비스/기능을 구현까지 |
| **PLAN** | Researcher → PM → 기획자 | 리서치 + PRD + 화면 기획까지 |
| **RESEARCH** | Researcher 단독 | 리서치·경쟁사 분석만 |
| **DESIGN** | 기획자 → 디자이너 | 기획안이 있고 디자인만 |
| **BUILD** | Engineer 단독 (또는 디자이너 → Engineer) | 디자인 스펙·문구가 있고 구현만 |
| **WRITE** | UX라이터 단독 | 문구 작업만 |
| **CUSTOM** | 사용자 정의 (Researcher·Engineer 포함) | 에이전트 직접 조합 |

- **플랫폼 분기**: 모든 모드는 `mobile / web / both`를 확정해 진행하며, 이 값에 따라 화면 설계·디자인(디바이스/브레이크포인트)·구현 기준이 달라집니다.
- **Figma**: 디자이너의 정본은 마크다운 스펙입니다. Figma 링크를 받으면 디자이너가 드로잉 플랜을 추가로 내고, 실제 Figma 작화는 메인 세션이 위임받아 그립니다(서브에이전트는 Figma 도구에 직접 접근하지 못하는 한계 때문). 링크가 없으면 마크다운 스펙만 정본으로 진행합니다.

## 실제 활용 사례

### 1. timefit — 다인원 회의 일정 조율 (0→1, 토스 디자이너 챌린지 2026 제출작)

"무가입 참석자가 가능/불가능 시간을 입력하면 최적 슬롯을 추천"하는 핵심 플로우를 0에서 설계해 **인터랙티브 웹 프로토타입으로 구현·배포**까지 마쳤습니다. 챌린지 성격상 PM 산출물은 답변 재료로만 재활용하고 무게중심을 디자이너로 옮긴, 모드를 실제 목적에 맞게 재편한 사례입니다.

### 2. timefit-flowtime — FULL 모드 7역할 전체 실행

같은 문제를 Researcher부터 Engineer까지 전 단계로 다시 실행한 사례. Reviewer 검수에서 PM·디자이너 산출물이 각각 `revise → pass`를 거쳤고, 1차 실행에서 쌓인 회사(flowtime) 결정·컨벤션이 자동 주입되어 적용됐으며, 디자인 스킬 게이트에서 `minimalist-ui`가 근거와 함께 채택됐습니다. **검수·기억·스킬 게이트가 한 실행 안에서 모두 작동한 검증 케이스**입니다.

### 3. github-curator — 비개발자를 위한 GitHub 도구 큐레이터

온보딩 + 데일리 카드 컨셉을 FULL 모드로 실행해 전 단계 `pass`, Figma 5화면 작화와 동작 코드까지 산출.

### 4. 위페어 — 실무 적용 (초기 적용 사례)

4역할 구성 시절, 위페어 파트너스 앱에서 신규 기능 기획·디자인과 기존 화면 UX 라이팅 감수에 실제로 적용했던 사례입니다. 이 시스템이 실험이 아니라 실무 맥락에서 출발했음을 보여줍니다.

## 앞으로의 계획

검증된 부분과 아직 설계/탐색 단계인 부분을 구분해 적습니다.

- **외부 모델 교차검토 (설계/탐색)** — 기획 단계에서 다른 모델의 의견을 선택적·반자동으로 받는 흐름. 아직 도입 전, 검증 후 결정 예정.
- **역할별 Definition of Done (탐색)** — KPI가 아니라 질적 완성 기준을 역할마다 정의해 검수 판정을 더 일관되게.
- **기억층 규모 검증** — 현재는 소수 프로젝트 검증. 더 많은 회사·프로젝트에 쌓이며 실제로 "똑똑해지는지" 지속 확인.
- **제품 조직 밖으로 확장** — 세일즈 제안서·마케팅 카피 등 인접 업무로의 확장 가능성 탐색.
