# Product AI Crew — 에이전트 구조 상세

> 7개 역할(Researcher~Engineer + Reviewer)의 역할·인풋·아웃풋 흐름과, 검수 루프·기억층·스킬 게이트·플랫폼 분기·Figma 위임을 정리한 문서입니다. 간결한 소개는 [README.md](README.md)를 참고하세요.

---

## 오케스트레이터 (Orchestrator)

사용자 요청을 받아 모드를 판단하고, 에이전트를 순서대로 호출해 전체 파이프라인을 관리합니다. 한 에이전트의 아웃풋을 다음 에이전트의 인풋으로 전달하며, 세션 상태를 `session.md`로 추적합니다.

작업 시작 시 오케스트레이터가 자동으로 하는 일:

1. **회사 식별 + 기억층 로딩** — 개인 프로파일(항상) + 해당 회사의 결정·컨벤션(있으면)을 `memory_context`로 묶어 모든 에이전트에 주입.
2. **페르소나 주입** — `workspace/[서비스명]/persona.md`가 있으면 `persona_path`로 함께 주입.
3. **플랫폼 확정** — `mobile / web / both` 중 확정해 PM→기획자→디자이너→Engineer 단계에 전달.
4. **모드 결정** — 아래 7개 모드 중 선택.

| 실행 모드 | 에이전트 호출 순서 | 사용 상황 |
| --- | --- | --- |
| **FULL** | Researcher → PM → 기획자 → 디자이너 → UX라이터 → Engineer | 신규 서비스/기능 전체 (구현까지) |
| **PLAN** | Researcher → PM → 기획자 | 리서치 + PRD + 화면 기획까지 |
| **RESEARCH** | Researcher 단독 | 리서치·경쟁사 분석만 |
| **DESIGN** | 기획자 → 디자이너 | 기획안이 있고 디자인만 |
| **BUILD** | Engineer 단독 (또는 디자이너 → Engineer) | 디자인 스펙·문구가 있고 구현만 |
| **WRITE** | UX라이터 단독 | 문구 작업만 |
| **CUSTOM** | 사용자 정의 | 에이전트 직접 조합 |

## 두 단계 품질 게이트

각 생성 에이전트가 산출물을 내면, 다음 에이전트로 넘어가기 전에 두 게이트를 차례로 통과해야 합니다.

1. **존재 검증 게이트** — 산출물 파일·필수 필드가 실제로 존재하는지 확인. 실패하면 `blocked` 처리 후 사용자 보고.
2. **Reviewer 검수** — 존재 검증을 통과한 직후, 내용 품질을 비평. `pass` / `revise` / `escalate` 판정.

```text
산출물 → [존재 검증 게이트] → [Reviewer 검수] → (pass면) 다음 단계
                  │ 실패              │ revise → 재작업(최대 2회) → 재검수
                  ▼                   │ escalate → 사용자 판단
               blocked
```

- `revise`: Reviewer의 검수 의견(`review_notes`)을 인풋에 포함해 해당 에이전트를 재호출. 재작업 횟수를 +1 하고 다시 두 게이트를 거칩니다.
- **재작업 상한 2회**: `revision_count >= 2`인데도 미흡하면 Reviewer가 `revise` 대신 `escalate`를 반환해 무한 루프를 막습니다.
- 검수 적용 범위는 모드를 따릅니다(FULL은 전 단계, RESEARCH·WRITE는 단독 산출물 1회 등). CUSTOM은 기본 on, 사용자가 끌 수 있습니다.

## Researcher

리서치를 수집·종합하고 경쟁/유사 서비스를 분석해, PM에게 넘길 "출처 기반 근거 요약"을 만듭니다. 추측이 아니라 출처에 근거합니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | 서비스명, 기능명, 기존 리서치 자료(있으면), 페르소나, 기억층 |
| 아웃풋 | 리서치 종합, key insights, 경쟁 분석 |
| 다음 에이전트 | PM |

> FULL/PLAN 진입 시, 사용자가 제공할 기존 리서치 자료(인터뷰·설문 등)가 있는지 먼저 확인합니다. 없으면 가용 범위(경쟁사 분석·웹 검색·페르소나)에서 수행하되 한계를 명시합니다.

## PM (Product Manager)

"무엇을 왜 만들 것인가"를 정의하고 제품 전략과 우선순위를 결정합니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | 리서치 종합·insights, 비즈니스 목표, 타겟 사용자, 플랫폼, 페르소나, 기억층 |
| 아웃풋 | PRD, 기능 목록, 우선순위(Impact×Effort), 브랜드 방향성 |
| 다음 에이전트 | 기획자 |

산출물: PRD · 기능 우선순위 정의서(Must / Should / Nice to Have) · 브랜드 방향성

## 기획자 (Product Planner)

"어떻게 만들 것인가"를 정의하고 화면 기획과 기능 명세서를 작성합니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | PRD, 기능 목록, 우선순위, 플랫폼, 브랜드 방향성, 페르소나, 기억층 |
| 아웃풋 | 화면 기획안, 기능 명세서, User Flow, 와이어프레임 |
| 다음 에이전트 | 디자이너 |

산출물: 화면 기획안 · 기능 명세서(트리거/동작/예외) · User Flow · **상태 정의(기본·로딩·에러·빈 상태)와 플랫폼별 예외**

## 디자이너 (Product Designer)

화면을 시각적으로 설계하고 Engineer가 바로 구현할 수 있는 컴포넌트 스펙을 정의합니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | 화면 기획안, 기능 명세서, 플랫폼, 브랜드 방향성, 승인된 디자인 스킬, Figma 링크(있으면), 페르소나, 기억층 |
| 아웃풋 | 디자인 스펙, 컴포넌트 스펙, (조건부) Figma 드로잉 플랜 |
| 다음 에이전트 | UX 라이터 |

산출물: 디자인 스펙(적용 스킬·전 화면/전 컴포넌트 명시) · 컴포넌트 스펙(default / hover / active / disabled / error) · figma 드로잉 플랜

디자이너 단계에는 본작업 전 두 개의 정지 지점이 있습니다.

- **Figma 링크 수령** — 디자이너 진입 직전 1회, 작화를 받을 Figma 페이지 링크를 묻습니다(없으면 '스킵').
- **디자인 스킬 선택 게이트** — 디자이너가 기획 결과물을 분석해 맞는 디자인 스킬을 근거와 함께 추천하고, **사용자가 고를 때까지 멈춥니다**. 승인된 스킬을 본작업에 강제 적용합니다.

> **정본은 마크다운 스펙입니다.** Figma는 링크를 받았을 때만 산출하는 시각화이지 스펙의 대체가 아닙니다. 또한 네이티브 서브에이전트는 Figma 도구에 직접 접근하지 못하므로, 디자이너는 드로잉 플랜만 내고 **실제 Figma 작화는 메인 세션이 위임받아** 토큰(Variables) → 컴포넌트·Variant → 전 화면 순으로 그립니다.

## Engineer

디자인 스펙과 문구를 받아 실제 동작하는 프론트엔드 코드로 구현합니다. **파이프라인의 마지막 단계이자 이 시스템의 최종 산출물**입니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | 디자인 스펙, 컴포넌트 스펙, Figma URL(있으면), 문구 시트, 기술 스택, 플랫폼 |
| 아웃풋 | 동작하는 프론트엔드 코드, 빌드 결과, 구현 노트 |
| 다음 에이전트 | 없음 (파이프라인 종료) |

산출물: 동작하는 프론트엔드 코드 · 빌드 검증 결과(pass) · 접근성 반영 · 미리보기 URL

> 범위는 프론트엔드 구현까지입니다. 백엔드·DB·API는 포함되지 않으며 데이터는 mock/정적으로 처리합니다.

## Reviewer

각 단계 산출물의 내용 품질을 비평하는 검수 에이전트입니다. 만드는 역할이 아니라, 존재 검증 게이트 통과 직후 호출되어 판정만 내리는 read-only 레이어입니다.

| 구분 | 항목 |
| --- | --- |
| 인풋 | 검수 대상 역할, 검수할 산출물 경로, 재작업 횟수, 페르소나 |
| 아웃풋 | `verdict`(pass / revise / escalate), 검수 의견(잘된 점 / 약한 점 / 막힌 지점) |
| 적용 | Researcher · PM · 기획자 · 디자이너 · UX라이터 · Engineer 전 단계 |

- **pass** → 통과, 다음 에이전트로 진행.
- **revise** → 검수 의견과 함께 해당 에이전트 재호출(최대 2회), 재작업 후 다시 검수.
- **escalate** → 작업 중단, 사유와 함께 사용자 판단으로 넘김.

## 프로젝트 간 기억층 (Memory Layer)

`session.md`가 한 프로젝트 *안*의 진행을 추적한다면, 기억층은 프로젝트 *사이*에 쌓이는 판단과 선호를 담아 시스템이 쓸수록 그 회사·그 디자이너에게 맞게 똑똑해지도록 만듭니다.

| 층 | 경로 | 담는 것 |
| --- | --- | --- |
| 개인 | `_memory/designer-profile.md` | 회사 불문 유지되는 작업 취향·원칙 |
| 회사 | `_memory/companies/[회사]/decisions.md` | 그 회사에서 내린 판단과 근거(왜) |
| 회사 | `_memory/companies/[회사]/conventions.md` | 그 회사의 톤·디자인 규칙·기술 제약 |

- **읽기 (자동 주입)**: 작업 시작 시 개인 프로파일 + 해당 회사 기억을 자동 로딩해 모든 에이전트에 주입합니다. 사용자가 매번 "지난번 결정 참고해"라고 말하지 않아도 적용됩니다.
- **쓰기 (반자동: 제안 → 승인)**: 자동 저장하지 않습니다(쓰레기 누적 방지). 기억할 가치가 있는 판단이 나오면 오케스트레이터가 제안하고, **사용자가 승인한 것만** 누적합니다.
- **담는 것**은 재현 가능한 판단의 근거·선호뿐입니다. 산출물(PRD·디자인·코드)·원본 데이터·세션 진행 상태는 담지 않습니다.

## 인풋/아웃풋 전체 흐름 (FULL 모드)

```text
[사용자 요청]
      ↓
[오케스트레이터] — 모드 판단 · 회사 식별 + 기억층 로딩 · 플랫폼 확정 · 세션 확인
      ↓        (페르소나 + 기억층을 이하 모든 단계에 자동 주입)
[Researcher]  아웃풋: 리서치 종합 · key insights · 경쟁 분석
      ↓  [존재 검증] → [Reviewer 검수] → pass
[PM]          아웃풋: PRD · 기능 목록 · 우선순위 · 브랜드 방향성
      ↓  [존재 검증] → [Reviewer 검수] → pass
[기획자]      아웃풋: 화면 기획안 · 기능 명세서 · User Flow · 상태 정의
      ↓  [존재 검증] → [Reviewer 검수] → pass
      ↓  ── Figma 링크 수령(1회) · 디자인 스킬 선택 게이트(사용자 승인) ──
[디자이너]    아웃풋: 디자인 스펙 · 컴포넌트 스펙 · (조건부) Figma 드로잉 플랜
      ↓  ── (Figma 링크 받았으면) 메인 세션이 위임받아 Figma 작화 ──
      ↓  [존재 검증] → [Reviewer 검수] → pass
[UX라이터]    아웃풋: UX 라이팅 가이드 · 화면별 문구 시트
      ↓  [존재 검증] → [Reviewer 검수] → pass
[Engineer]    아웃풋: 동작하는 프론트엔드 코드 · 빌드 결과 ★최종 산출물
      ↓  [존재 검증] → [Reviewer 검수] → pass
[작업 완료 보고]  — session.md status: complete · 기억 제안(있으면)
```

> 검수 루프 표기: 각 단계의 `[Reviewer 검수]`는 `revise` 시 해당 에이전트로 되돌아가 재작업(최대 2회)하고, `escalate` 시 사용자 판단으로 넘어갑니다. 위 흐름은 모두 `pass`인 경우의 직선 경로입니다.
